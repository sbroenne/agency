# Forge: Agent Skill Authoring & Distribution

Forge is a structured system for creating, tracking, and publishing reusable agent skills for the Squad ecosystem. This document describes skill types, distribution patterns, and release workflows.

## Overview

Forge helps teams author and organize agent skills into two categories:
- **Library skills** — reusable utilities without UI (tools-only packages)
- **Customer-facing skills** — complete workflows with agents and UI (skills + agents + prompts)

Forge distinguishes between **dev repos** (where skills are authored and tested) and **published skill repos** (where they are versioned and released).

## Terminology

When discussing Forge:
- Use **Agent Skill** as the primary unit of reuse
- Use **skill distribution** or **package** for how skills are organized
- Use **`plugin.json`** only when referencing the packaging format
- Do **not** use "Forge plugin" — this term is not a core product concept

## Local Working Repo Model

Forge knowledge stays in `agency`, but authoring happens in a working repo on the user's machine once implementation starts.

- `agency` keeps the rules, scaffolds, workflows, and reference catalog.
- Forge can copy or adapt the `squads/forge/` reference assets into a local working repo.
- That working repo is where plugin-specific code, tests, and release automation evolve.

For this repository's lean starter implementation, the tracked reference catalog lives at `squads/forge/home/catalog.json`.

## Repo Topology

### Dev Repo Structure
Skills are developed in the dev repo with an explicit directory structure:

```
dev-repo/
├── skills/
│   ├── skill-a/
│   │   ├── plugin.json          # Skill manifest
│   │   ├── index.js             # Skill implementation
│   │   ├── schema.json          # Tool schema
│   │   ├── package.json         # Dependencies
│   │   └── README.md
│   └── skill-b/
│       └── ...
├── package.json                 # Workspace root
├── PLUGINS.md                   # Registry (generated/tracked)
└── tooling/
    ├── scripts/
    │   ├── build-skills.mjs     # Builds manifests
    │   └── validate-skills.mjs  # Validates structure
    └── tests/
        └── skills.test.mjs      # Skill tests
```

### Published Skill Distribution Structure
Published skill distributions are released as standalone npm packages (or Git repos) with:

```
published-skill-repo/
├── package.json
├── skill-a/                     # For library skills
│   ├── index.js
│   └── schema.json
├── skill-b/
│   └── ...
├── agents/                      # For customer-facing skills only
│   ├── agent-name/
│   │   └── index.js
│   └── ...
├── prompts/                     # For customer-facing skills only
│   ├── system-prompt.txt
│   └── ...
├── README.md
└── CHANGELOG.md
```

## Skill Registry (PLUGINS.md)

The dev repo includes `PLUGINS.md` — a human-readable registry of all skills being developed. It acts as:
- **Snapshot** of plugin inventory
- **Status indicator** — which plugins are ready for release
- **Change history** — when plugins are added, deprecated, or updated

### Format

```markdown
# Plugins Registry

> Last updated: 2026-03-20 by CI pipeline

## Library Plugins

| Name | Version | Status | Published | Maintainer |
|------|---------|--------|-----------|-----------|
| plugin-a | 0.1.0 | dev | — | Team A |
| plugin-b | 1.0.0 | stable | npm | Team B |

## Customer-Facing Plugins

| Name | Version | Status | Published | Maintainer |
|------|---------|--------|-----------|-----------|
| plugin-x | 0.2.0 | beta | github-releases | Team X |
| plugin-y | 2.1.0 | stable | npm | Team Y |

## Deprecated

| Name | Deprecated | Reason |
|------|-----------|--------|
| old-plugin | 2026-03-15 | Superseded by plugin-c |
```

In larger dev repos, `PLUGINS.md` is often generated by `npm run build:plugins`. Lean teams can maintain it directly until automation becomes necessary.

## Plugin Classification

### Library Plugins
- **Contents**: Skills only (no agents, no UI)
- **Use case**: Reusable, stateless utilities (e.g., Excel automation, API clients, formatters)
- **Publishing**: npm or private registry
- **Dependencies**: Minimal; no framework dependencies
- **Examples**: 
  - `excel-mcp-server` (MCP server for Excel operations on Windows)
  - `@myorg/skill-http-client`
  - `@myorg/skill-json-parser`

### Customer-Facing Plugins
- **Contents**: Skills + agents + prompts/system instructions
- **Use case**: Complete, opinionated workflows (e.g., code review bot, docs generator)
- **Publishing**: Private registry or gated distribution
- **Dependencies**: May depend on Copilot CLI or squad-cli
- **Example**: `@myorg/agent-reviewer`, `@myorg/agent-docs-writer`

## Agency-to-Working-Repo Workflow

1. **Start from Forge guidance** in `agency`
2. **Create the local working repo** and copy/adapt the relevant scaffold
3. **Validate locally** — run the working repo's checks
4. **Publish npm/release** — package is published (npm, GitHub releases, etc.)
5. **Update Forge tracking surfaces** — refresh `squads/forge/PLUGINS.md` and `squads/forge/home/catalog.json` when you want discoverability in Agency

## Release Workflow

### Before Release
- [ ] Plugin passes validation: `npm run validate:plugins`
- [ ] Tests pass: `npm test`
- [ ] CHANGELOG.md updated with release notes
- [ ] Version bumped in `plugin.json` and `package.json`
- [ ] `PLUGINS.md` reflects new version and status

### Release Steps
1. Create a Git tag: `git tag plugin-a@0.1.0`
2. Push tag and trigger release workflow
3. CI publishes to npm/registry
4. Agency's Forge registry entry is updated with the published URL

### After Release
- [ ] Agency's Forge registry shows the new version
- [ ] Changelog is accessible (GitHub releases, npm, etc.)
- [ ] Downstream users can discover and install

## Architecture Recommendation Flow

When authoring a new Forge plugin, Forge asks:

**Q: What does this plugin do?**
- Single reusable capability (e.g., "format JSON code") → **Library plugin**
- Complete workflow with agent behavior (e.g., "review code") → **Customer-facing plugin**

**Q: Does it need UI/prompts?**
- No → **Library plugin**
- Yes → **Customer-facing plugin** → Does it need APM (Advanced Plugin Management)?
  - Simple prompts/instructions → Skip APM
  - Complex state/routing → Introduce APM

This keeps plugin authorship lean — APM only when required.
